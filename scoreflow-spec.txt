1. Resumo Executivo
1.1 Português
O ScoreFlow é uma aplicação web que transforma arquivos de áudio (MP3 ou WAV) em partituras digitais no formato MusicXML. O fluxo principal é simples: o usuário faz upload de um único arquivo de áudio, o backend processa automaticamente usando Magenta (conversão áudio → MIDI) e music21 (conversão MIDI → MusicXML), e o resultado é entregue com pré-visualização da partitura e opções de download. O público-alvo principal são músicos, compositores, estudantes de música, produtores e professores que precisam converter gravações em notação musical de forma rápida, sem conhecimento técnico avançado. O produto prioriza simplicidade, clareza de feedback durante o processamento e qualidade da transcrição.
1.2 English
ScoreFlow is a web application that transforms audio files (MP3 or WAV) into digital sheet music in MusicXML format. The core flow is simple: the user uploads a single audio file, the backend processes it automatically using Magenta (audio-to-MIDI) and music21 (MIDI-to-MusicXML), and the result is delivered with a score preview and download options. The primary target audience includes musicians, composers, music students, producers, and teachers who need to convert recordings into musical notation quickly, without advanced technical knowledge. The product prioritizes simplicity, clear processing feedback, and transcription quality.

3.2 Fluxos Alternativos e de Erro
Arquivo inválido: usuário recebe mensagem inline imediata, área de upload permanece ativa.
Falha no upload (rede): retry automático (3x), depois mensagem com botão de tentar novamente.
Timeout no processamento: após 5 minutos sem progresso, notifica usuário com opção de cancelar ou aguardar.
Erro no Magenta: mensagem específica (&quot;Não foi possível transcrever este áudio&quot;) + sugestão (tentar outro arquivo, mono, melhor qualidade).
Erro no music21: mensagem (&quot;Erro na conversão para partitura&quot;) + disponibiliza MIDI parcial para download.
Usuário tenta novo upload durante processamento: modal de confirmação (&quot;Cancelar processamento atual?&quot;). Se sim, cancela job e aceita novo arquivo.

4. Componente de Upload
4.1 Regras Funcionais
Aceita exatamente 1 arquivo por vez. Se o usuário arrastar múltiplos, aceitar apenas o primeiro e exibir aviso.
Formatos aceitos: .mp3, .wav (MIME types: audio/mpeg, audio/wav, audio/x-wav, audio/wave).
Tamanho máximo: 200 MB. Justificativa: cobre gravações WAV de até ~18 minutos em 44.1kHz/16bit stereo, suficiente para a grande maioria das peças musicais.
Duração máxima recomendada: 10 minutos (aviso soft, não bloqueia). Justificativa: transcrições longas perdem precisão e aumentam custo computacional.
Botão de upload desabilitado durante processamento ativo (spinner no botão).
Drag &amp; drop ativo: área destaca-se com borda tracejada e cor de accent ao receber hover de arquivo.

4.4 UX: Novo Upload Durante Processamento
Quando o usuário tenta enviar um novo arquivo enquanto há um job ativo:
O componente de upload fica visualmente desabilitado (opacidade 50%, cursor not-allowed).
Se o usuário tentar arrastar um arquivo, aparece um tooltip: &quot;Aguarde o processamento atual&quot; / &quot;Wait for current processing&quot;.
Um link discreto &quot;Cancelar processamento atual&quot; / &quot;Cancel current processing&quot; fica visível abaixo da barra de progresso.
Ao clicar, abre modal de confirmação: &quot;Tem certeza? O processamento atual será cancelado.&quot; / &quot;Are you sure? Current processing will be cancelled.&quot;
Após confirmação, o job é cancelado via endpoint, a UI volta ao estado idle e aceita novo upload.

5. Barra de Progresso — Linha de Processamento
5.1 Design &amp; Animação
A barra de progresso é o elemento central de feedback durante o processamento. Deve comunicar confiança, não ansiedade.

Layout: barra horizontal segmentada em 5 fases, com ícones circulares conectados por linhas. Estilo &quot;stepper&quot; (similar a checkout de e-commerce, mas com estética musical).
Fase ativa: ícone pulsa suavemente (scale 1.0 → 1.05, ease-in-out, 1.5s loop). Linha entre fases anteriores e ativa fica preenchida com gradiente (accent color). Linha futura fica cinza claro.
Fase completa: ícone recebe checkmark animado (stroke-dashoffset), cor sólida de sucesso (verde suave).
Texto dinâmico: abaixo da barra, texto curto do estado atual + microcopy descritivo (rotaciona a cada 3s com fade transition).
Duração estimada: se o backend fornecer estimativa, exibir &quot;Tempo estimado: ~2 min&quot; / &quot;Estimated time: ~2 min&quot;. Se não, exibir animação de pontos (&quot;...&quot;).
Erro em fase específica: ícone da fase muda para ícone de erro (X vermelho), texto exibe mensagem de erro, botão de retry aparece.

6. Tela de Resultado
6.1 Layout e Visualização
A tela de resultado é o momento de &quot;recompensa&quot; do usuário. Deve ser visualmente satisfatória.

Preview da Partitura: renderização do MusicXML usando OpenSheetMusicDisplay (OSMD) ou VexFlow. A partitura é exibida em um container responsivo com scroll horizontal se necessário. Fundo branco (mesmo em dark mode) para simular papel de partitura. Zoom control (+/–) disponível.
Player inline (Fase 2): barra de playback sincronizada com a partitura — destaca a nota/compasso atual durante reprodução MIDI.
Informações do arquivo: card com nome original, duração, tonalidade detectada, compasso, número de notas, tempo estimado (BPM).

10.2 Recomendações
Contraste mínimo WCAG AA (4.5:1 para texto normal, 3:1 para texto grande).
Ícones: usar versões outline em light, filled em dark (melhor visibilidade).

Transição entre temas: fade suave de 200ms em todo o body.

11. Design Estético — Moodboard Verbal
11.1 Conceito Central
&quot;Onde a música encontra a tecnologia.&quot; O ScoreFlow deve transmitir sofisticação, precisão e calor humano. Não é um dashboard de DevOps — é uma ferramenta para artistas. O visual deve evocar a sensação de abrir uma partitura nova enquanto celebra o processo tecnológico que a criou.

11.2 Elementos Visuais Recorrentes
Ondas sonoras estilizadas como elementos decorativos (backgrounds, dividers, loading).
Pentagramas e claves como motifs gráficos sutis (não literal, abstrato).
Piano roll fragments como padrão de textura em cards e headers.
Linhas de tempo / timelines como metáfora visual para o processamento.
Partículas sutis: pontos flutuantes que simulam notas musicais em transições.
11.3 Tipografia
Headlines: Inter, Satoshi ou Space Grotesk (geométrica, moderna, excelente legibilidade).
Body: Inter ou system-ui (para performance e familiaridade).
Partitura/código: JetBrains Mono ou Fira Code (monospace, para exibir nomes de arquivo/log).
Tom de voz: profissional mas acolhedor. Fala COM o músico, não SOBRE tecnologia. Evita jargão de ML. Usa metáforas musicais quando possível.

11.4 Três Variações de Estilo

Variação A: Minimal
Limpo, muito espaço em branco, tipografia como protagonista. Inspiração: Notion, Linear, Apple Music (seção clássica). Ícones line-weight fino. Sem gradientes. Animações sutis (opacity, translate).
Paleta 1: #FFFFFF, #F7F9FC, #1A1A2E, #3B82F6 (azul vibrante), #10B981 (verde menta).
Paleta 2: #FAFAFA, #F0EDE8, #1C1917, #7C3AED (violeta), #F59E0B (âmbar).

Variação B: Neo-Tech
Dark-first, gradientes sutis, glassmorphism leve. Inspiração: Spotify, Vercel, GitHub Dark. Borda luminosa em elementos ativos. Efeito blur em overlays. Animações com spring physics.
Paleta 1: #0F0F1A, #1A1A2E, #E8E8E8, #00D4FF (ciano neon), #FF6B9D (rosa coral).
Paleta 2: #0D1117, #161B22, #C9D1D9, #58A6FF (azul GitHub), #3FB950 (verde GitHub).

Variação C: Retro-Analog
Quente, texturas de papel, tipografia serifada para headers. Inspiração: capas de vinil, estúdios analógicos, VU meters. Ícones com cantos arredondados, sombras suaves. Animações com easing &quot;bouncy&quot;.
Paleta 1: #FFF8F0, #F5E6D3, #2D1B0E, #C05621 (terracota), #2B6CB0 (azul vintage).
Paleta 2: #FFFDF7, #E8DDD3, #1A202C, #9B2C2C (vermelho vinícola), #2C7A7B (teal retrô).

Recomendação: para o MVP, começar com Variação A (Minimal) — menor custo de implementação, melhor acessibilidade, fácil de escalar. Variação B pode ser adotada em Fase 2 como tema alternativo.

12. Acessibilidade
12.1 Checklist Mínimo (MVP)
Contraste: todos os textos atendem WCAG AA (4.5:1 normal, 3:1 grande). Verificar com Lighthouse/axe.
Labels: todos os inputs, botões e áreas interativas têm aria-label ou label associado.
Navegação por teclado: tab order lógico, focus visible em todos os elementos interativos, atalhos (Enter para upload, Escape para fechar modais).
Leitores de tela: aria-live regions para atualizações da progress bar (announcements de cada estado). Role=&apos;progressbar&apos; com aria-valuenow.
Drag &amp; drop: alternativa por botão de seleção de arquivo (nunca depender só de drag).
Animações: respeitar prefers-reduced-motion — desativar pulsações e partículas, manter transições instantâneas.
Idioma: atributo lang correto no HTML (&quot;pt-BR&quot; ou &quot;en&quot;) atualizado dinamicamente.
Mensagens de erro: lidas automaticamente por screen readers via aria-live=&quot;assertive&quot;.
12.2 Player / Piano Roll (Fase 2)
Player de áudio com controles nativos (play/pause/seek) acessíveis por teclado.
Piano roll: fornecer descrição textual alternativa (&quot;Partitura com 48 notas, compasso 4/4, tonalidade de Dó maior&quot;).
Notas na partitura renderizada: se interativas (Fase 2, edição), cada nota deve ser focable e descrita.
Preferência de alto contraste na partitura: opção de renderizar com linhas mais grossas e notas maiores.

13.3 Fila de Jobs e Status
Cada job possui um ID único (UUID v4).
Status armazenado em Redis com TTL de 24h (cleanup automático).
Client faz polling a cada 2s no endpoint de status. Justificativa: simples no MVP; Fase 2 pode migrar para WebSocket/SSE.
Cada worker atualiza o status no Redis ao iniciar e finalizar sua etapa.
Se o worker falha, o status é marcado como &apos;error&apos; com mensagem específica da etapa.

14. Políticas de Privacidade &amp; Retenção (LGPD)
14.1 Aviso no Upload
Texto exibido discretamente abaixo da área de upload, antes do botão de enviar:

PT: &quot;Ao enviar seu arquivo, você concorda com nossos Termos de Uso. Seus dados são processados exclusivamente para gerar a partitura e serão excluídos automaticamente em 7 dias. Você pode excluir seus dados a qualquer momento nas Configurações.&quot;
EN: &quot;By uploading your file, you agree to our Terms of Service. Your data is processed solely to generate the score and will be automatically deleted within 7 days. You can delete your data at any time in Settings.&quot;

14.2 Política de Retenção
Arquivos de áudio originais: excluídos após processamento (imediatamente após conversão). Justificativa: não há razão para reter o áudio original após o pipeline completar.
Arquivos de resultado (MusicXML, MIDI, log): retidos por 7 dias. Usuário pode baixar nesse período.
Metadados do histórico: retidos por 30 dias (apenas nome, data, status — sem conteúdo do arquivo).
Usuário pode: excluir itens individuais, limpar todo histórico, solicitar exclusão total da conta.
LGPD / compliance: fornecer &quot;Baixar meus dados&quot; (Fase 2) para atender ao direito de portabilidade.
14.3 Botão de Exclusão Imediata
Presente em cada item do histórico e nas Configurações. Fluxo: click → modal de confirmação → exclusão via endpoint DELETE → feedback visual (item some com animação fade).

20. Sugestões Adicionais / Inovações
20.1 MVP
Detecção automática mono/stereo: se stereo, oferecer conversão automática para mono antes da transcrição (melhora qualidade significativamente para instrumentos solo). Custo zero de implementação, alto impacto na qualidade.
&quot;Confidence score&quot; por nota: o Magenta Onsets and Frames retorna probabilidades. Exibir notas de baixa confiança com cor diferente na partitura (ex.: cinza claro). Ajuda o usuário a saber o que revisar.

20.2 Fase 2
Separação de stems prévia: integrar Demucs ou Spleeter para separar instrumento antes da transcrição. Permite transcrição por stem (vocal, guitarra, bateria, baixo). Melhoria dramática na qualidade de polifonia.
Comparação A/B: permitir ao usuário comparar dois resultados de reprocessamento lado a lado.
Webhook / Callback: para usuários de API (Fase 2), notificar via webhook quando o processamento terminar.
Onboarding interativo: primeira vez do usuário, mostrar tooltips guiados (&quot;Arraste um áudio aqui&quot;, &quot;Aqui você acompanha o progresso&quot;). Reduz fricção e abandono.
Collaborative playlists: professores podem enviar múltiplos áudios (batch) e compartilhar resultados com alunos via link.

20.3 Opcional / Exploração
Basic Pitch como alternativa: modelo da Spotify (MIT license) para pitch detection. Mais leve, mais rápido, excelente para instrumentos mono—pode ser oferecido como modo &quot;rápido&quot; ao lado do Magenta.
Export para Musescore/Finale: além de MusicXML, oferecer .mscz (Musescore nativo) para fluxo mais direto.
Feedback loop: usuários reportam notas erradas na partitura. Esses dados podem ser usados para fine-tuning futuro do modelo.
PWA + offline: permitir que o usuário instale como app e use offline com magenta.js (client-side, limitado).
Integração com DAWs: plugin VST/AU que envia áudio diretamente para o ScoreFlow e recebe MIDI de volta.
